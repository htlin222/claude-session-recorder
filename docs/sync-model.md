# 旁白與畫面的同步模型（演化史）

「讓語音跟終端動作對得上」是這支 demo 最難、改最多次的部分。
記錄四個版本與各自為何被換掉，免得未來又走回頭路。

## v1 — J-cut，命令先跑、旁白後到 ❌
每段開頭就把命令打完並執行，旁白用 J-cut（聲音提前 0.8s 進場）蓋在上一段尾巴。
**問題**：命令在旁白還沒講完就執行了，旁白對著靜止的輸出念十幾秒；
J-cut 又讓聲音更早，整體「聲畫完全對不上」。
**教訓**：J-cut 適合剪輯轉場，不適合「邊講邊操作」的教學。

## v2 — 先講解、句末才觸發 ✅（方向對了）
每段重構成：打出命令（不執行）→ 旁白講解 → **句子一結束就 Enter** → 出結果。
關鍵：tape 的「講解 Sleep」長度 = 實測旁白時長，所以旁白結尾和 Enter 對齊。
**教訓**：核心問題是**結構**（先講再做），不只是微調時間。

## v3 — 統一五拍 ⚠️（太碎）
把每個命令拆成：引入旁白 → 打字 → 解釋命令 → `<CR>` → 解釋輸出，
每拍一段獨立 edge-tts。**問題**：旁白被切成一小段一小段，段間有停頓，
聽起來斷斷續續。使用者明確不要這種。

## v4 — 一條連續旁白，動作釘在句界 ✅✅（最終）
整支影片只合成**一段**連續 edge-tts 語音；終端的打字 / `<CR>` / 輸出
都釘在這段語音的**句子邊界**上（用 edge-tts 的逐句時間戳）。
聲音從頭到尾不斷，命令在「講到那句」時自然觸發。

```
clear(hidden) → 引入句(淨屏) → 打字(逐 token) → 解釋句結束=Enter → 輸出 → 描述句 → 下一段
                └──────────────── 同一段連續語音，不間斷 ────────────────┘
```

### 為什麼可行（也是關鍵設計）
- **動作跟著「打字」走，不跟著「語音逐字」走**：appear-on-type。
  只要打字節奏可控，就不需要逐詞時間戳。
- **Enter 釘在句界**：解釋句結束 = Enter，下一句正好開始描述輸出，無縫銜接。
- **剛性音檔的漂移**用 `atempo` 校正（見 lessons-learned）。

## v5 微調 — Hero 命令逐 token 慢打
v4 的命令是一次 `Type` 打完（35ms/char，~1.5s 太快）。
改成**逐 token 打字**：打一個 flag、停 `TOKEN_STEP=0.6s`、再打下一個，
每個 token 打完的瞬間右側側欄就浮出該註解。
同時**旁白改寫成依打字順序逐一念 flag**（「用 -a 封存、-v 顯示細節，再加上 --delete」），
讓語音、打字、側欄三者順序一致。
解釋句要寫得夠長（~4-5s）以容納 ~3.2s 的慢打。

## 可調參數（build.py）
- `TOKEN_STEP` — 每個 token 前的停頓（慢打節奏）。
- `OUTPUT_MIN` — Enter 後在輸出上的最小停留。
- 解釋句長度 — 直接改 `SCRIPT` 裡那句旁白的字數。
- `RATE` — edge-tts 語速（現 `+0%`）。
